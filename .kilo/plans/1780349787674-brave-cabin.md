# SDD Review: Inconsistencies, Ambiguities & Open Questions

## 1. Critical Inconsistencies (Will break implementations)

### 1a. Spec filename mismatch
- `README.md` line 15 & `CONTRIBUTING.md` line 12 reference `docs/specs/betting-engine.md`, but the actual file is named `better-engine.md`.
- `SPECS.md` line 16 also references `specs/betting-engine.md`.
- **Action:** Rename or unify. The file is currently `better-engine.md` — likely a typo for `betting-engine.md`.

### 1b. Frontend framework contradiction
- `README.md`, `SPECS.md`, and `CONTRIBUTING.md` all specify **Angular 17+ (Zoneless, Signals)**.
- `auth-sessions.md` line 9 explicitly says "The **Svelte** frontend executes the OAuth2 flow".
- `market-sync.md` line 17 says "When the **Svelte** frontend detects the X-Cache-Status header".
- **Action:** Standardize all references to "Angular frontend".

### 1c. Missing database tables referenced in specs
- `auth-sessions.md` lines 23-31 defines a `revoked_tokens` table and says "To be added to database-schema.md" — **but it is not present** in `database-schema.md`.
- `better-engine.md` line 38 references a `historical_bets` ledger table — **not defined** in `database-schema.md`.
- **Action:** Add `revoked_tokens` and `historical_bets` DDL to `database-schema.md`, or remove references if not needed.

### 1d. Portfolio schema vs. betting spec mismatch
- `better-engine.md` line 30 references a `bet_type` column ('LONG' or 'SHORT') in the `portfolios` table.
- `database-schema.md` `portfolios` table definition has no `bet_type` column.
- **Action:** Add the `bet_type` column, or remove the reference and redesign.

## 2. Ambiguities (Need clarification before SDD is complete)

### 2a. "Absolute Zero Idle Cost" — unscoped
- Cloudflare Workers have cold-start latency; D1 has storage costs ($0.25/GB/month). The constitution does not define what "idle" means or what acceptable cold-start latency is.
- Clarify: Are cron-scheduled workers (Workers Paid Plan) acceptable for hydration, or must hydration be 100% on-demand?

### 2b. External financial API contract
- No specification for which external market data provider (e.g., Alpha Vantage, Finnhub, Yahoo Finance, Polygon.io) is used.
- No error-handling, rate-limit, or fallback strategy defined when the financial API fails.
- **Action:** Add a section to `market-sync.md` or a new spec for external API contracts.

### 2c. Authentication flow gap
- `auth-sessions.md` references `AUTH_SECRET` but no key-rotation, rotation schedule, or secret-management strategy (e.g., Cloudflare Secrets vs D1-stored key) is specified.
- No definition of the internal JWT claims structure (exp, sub, iat, jti, etc.) or TTL.
- **Action:** Add JWT payload schema, TTL, and secret-management section to `auth-sessions.md`.

### 2d. Floating-point currency precision
- `database-schema.md` uses `REAL` for all monetary values. `better-engine.md` explicitly calls out "floating-point checks" but does not specify rounding, precision, or whether integer (cents) representation is needed.
- **Action:** Specify currency precision strategy (e.g., store as integer cents, or define max decimal places).

### 2e. Market hydration cycle gaps
- No specification of **which** assets are tracked or how the tracked-ticker list is managed (no `tracked_assets` config table).
- `asset_history` has daily granularity, but the betting engine evaluates on every cash-out — is the live price intraday or end-of-day? If intraday, how is it refreshed?
- **Action:** Define ticker discovery/management mechanism and price-update frequency.

### 2f. Monorepo tooling undefined
- README mentions npm install/dev but does not specify package.json workspaces, Turborepo, Nx, or pnpm/NPM workspaces to manage the monorepo.
- No ESLint/TypeScript config root location defined for shared linting across packages.
- **Action:** Add monorepo tooling section to `SPECS.md` or `CONTRIBUTING.md`.

### 2g. Missing API contract specification
- No OpenAPI/Swagger spec or typed contract (e.g., tRPC, Zod schema, OpenAPI YAML) for frontend-backend interaction.
- `auth-sessions.md` references endpoints (`/api/v1/auth/callback`, `/api/v1/auth/logout`) but no comprehensive route list exists.
- **Action:** Add an API contract spec or reference it in `SPECS.md`.

### 2h. CORS/rate-limiting/security
- No CORS policy specified for the Hono API.
- No rate-limiting specification (even for the paid Workers plan).
- No CSRF, XSS, or injection-prevention measures documented for the Angular client or Hono API.

## 3. Open Questions (Blocking SDD finalization)

1. **What is the external market data provider?** (Section 2b)
2. **How is the ticker list curated and updated?** (Section 2e)
3. **Is `real-time` intraday price data required, or is end-of-day sufficient for game logic?** (Section 2e)
4. **What is the JWT TTL and refresh strategy?** (Section 2c)
5. **Should the `revoked_tokens` table use an index on `user_id` for fast session invalidation lookups?** (Currently only indexed on `expires_at`)
6. **Should the `historical_bets` table be a ledger with UPDATE-able status, or append-only?**
7. **What is the acceptable cold-start latency target for the "on-demand" hydration principle?** (Section 2a)
8. **Is the `bet_type` column on `portfolios` a required v1 feature, or can shorts be added later?**
9. **Will there be a Cloudflare D1 migration tool (Drizzle Kit), and where are migration files stored?**
10. **What frontend build/deploy toolchain is used?** (Angular CDK + Cloudflare Pages adapter is not a standard combo — is `@cloudflare/angular` or a custom adapter planned?)
